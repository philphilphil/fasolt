## Per-Area Summaries

### Dependencies — [🔴 0 / 🟠 3 / 🟡 5 / 🔵 4]

No critical vulnerabilities. The main risks are: picomatch ReDoS in the npm transitive tree (dev-only), OpenIddict 4 minor versions behind (security-critical OAuth library), and NuGet lock files not enforced in CI. The project has no automated dependency update tooling (Dependabot/Renovate). Positives: all lock files committed, npm integrity hashes present, iOS has zero third-party dependencies.

### Auth & Authorization — [🔴 2 / 🟠 3 / 🟡 5 / 🔵 3]

The two critical findings are missing CSRF tokens on server-rendered OAuth login and consent forms. High-severity issues include a missing production email sender (password reset broken in prod), an open redirect on consent denial, and unbounded dynamic client registration. Tenant isolation is excellent — every data query filters by userId. PKCE is enforced, cookies are properly secured, and admin endpoints require cookie-only auth.

### Injection & Input Validation — [🔴 0 / 🟠 2 / 🟡 5 / 🔵 4]

No SQL injection, XSS, command injection, or path traversal vectors found. The high-severity findings are LIKE wildcard injection in search (DoS vector) and missing length limits on card text fields (storage abuse). The SVG sanitizer is well-designed with server-side and client-side defense-in-depth. All EF Core queries are parameterized. DOMPurify is used consistently for HTML rendering.

### Data Exposure & Cryptography — [🔴 0 / 🟠 3 / 🟡 4 / 🔵 4]

DevEmailSender logs password reset tokens at Information level. The admin API exposes internal ASP.NET Identity GUIDs (rest of app uses NanoIds). User emails are embedded in database log records (GDPR risk). OpenIddict certificates are loaded without password protection. Positives: public ID pattern for cards/decks, HttpOnly cookies, generic error messages in production, proper .gitignore coverage, production DB password externalized.

### Code Quality — [🔴 0 / 🟠 3 / 🟡 5 / 🔵 4]

Silent error swallowing in CardsView's addCardToDeck, unsafe `undefined as T` cast in the API client, and duplicated notification queries are the high-severity items. Several Pinia stores silently swallow fetch errors leaving stale state. CardTable.vue uses `any` types throughout. The codebase has significant test coverage gaps: OAuthEndpoints (564 lines, 0 tests), AdminEndpoints, AccountEndpoints, and the frontend auth store are all untested.

### Architecture — [🔴 0 / 🟠 4 / 🟡 4 / 🔵 3]

Three endpoint files (ReviewEndpoints, NotificationEndpoints, AdminEndpoints) bypass the service layer and contain business logic with direct DbContext access. SearchService has a circular dependency on the API layer. The Domain layer has no interfaces or value objects — card state is a plain string. OAuthEndpoints is a 560-line monolith mixing HTML rendering, OAuth protocol logic, and consent handling. Positives: clean endpoint pattern, excellent MCP-to-service code reuse, solid frontend and iOS architecture.

### Configuration & Infrastructure — [🔴 0 / 🟠 3 / 🟡 5 / 🔵 3]

Container runs as root, no health check for the app container in production, and CI workflow lacks least-privilege permissions. Dev launch profile binds to 0.0.0.0 (exposes dev server on all interfaces). Missing Permissions-Policy and COOP headers. No request body size limit configured. Positives: database bound to localhost in dev, network-isolated in prod, multi-stage Docker build, proper security headers (CSP, HSTS, X-Frame-Options), rate limiting configured.

## Positive Patterns Observed

- **Excellent tenant isolation** — Every data-access query consistently filters by `userId` across all services, endpoints, and MCP tools. No cross-user data access is possible.
- **Defense-in-depth SVG sanitization** — Server-side allowlist-based sanitizer + client-side DOMPurify with strict configuration.
- **No injection vectors** — All EF Core queries are parameterized. No raw SQL, command injection, or path traversal risks.
- **Solid cookie security** — HttpOnly, SameSite=Strict, SecurePolicy=Always in production. No auth tokens in localStorage.
- **PKCE enforced** — OAuth flow requires Proof Key for Code Exchange, preventing authorization code interception.
- **Admin endpoints restricted to cookie-only auth** — OAuth bearer tokens cannot access admin functionality.
- **Good MCP-to-service reuse** — MCP tools delegate to Application services rather than reimplementing logic.
- **iOS security best practices** — Keychain with `kSecAttrAccessibleWhenUnlockedThisDeviceOnly`, token refresh coalescing, zero third-party dependencies.
- **Rate limiting on all endpoint groups** — Three tiers (auth-strict, auth, api) with appropriate limits.
- **Security headers present** — CSP, X-Content-Type-Options, X-Frame-Options (DENY), Referrer-Policy, HSTS all configured.
- **Public ID pattern** — Cards and decks use NanoIds in API responses, not internal database GUIDs.
- **Cursor-based pagination** — Proper scalable pagination pattern for card listing.
