### `INFRA-H001` â€” Container Runs as Root
- **ID:** `INFRA-H001`
- **File:** `Dockerfile:L18-L26`
- **Risk:** The runtime container runs as root. If the application is compromised (e.g., via RCE), the attacker has root privileges inside the container, enabling container escape or lateral movement.
- **Evidence:**
  ```dockerfile
  FROM mcr.microsoft.com/dotnet/aspnet:10.0
  RUN apt-get update && apt-get install -y --no-install-recommends libkrb5-3 && rm -rf /var/lib/apt/lists/*
  WORKDIR /app
  COPY --from=backend /app/publish ./
  COPY --from=frontend /app/dist ./wwwroot/
  # No USER directive â€” defaults to root
  ENTRYPOINT ["dotnet", "fasolt.Server.dll"]
  ```
- **Fix:** Add a non-root user to the runtime stage:
  ```dockerfile
  FROM mcr.microsoft.com/dotnet/aspnet:10.0
  RUN apt-get update && apt-get install -y --no-install-recommends libkrb5-3 && rm -rf /var/lib/apt/lists/*
  RUN addgroup --system fasolt && adduser --system --ingroup fasolt fasolt
  WORKDIR /app
  COPY --from=backend /app/publish ./
  COPY --from=frontend /app/dist ./wwwroot/
  USER fasolt
  ENTRYPOINT ["dotnet", "fasolt.Server.dll"]
  ```

### `INFRA-H002` â€” No Container Health Check in Dockerfile or Production Compose
- **ID:** `INFRA-H002`
- **File:** `Dockerfile` and `docker-compose.prod.yml`
- **Risk:** The `app` service in production has no health check. Orchestrators (Docker, Traefik) cannot distinguish a hung process from a healthy one, meaning traffic may be routed to an unresponsive container indefinitely. The database service has a health check, but the application container does not.
- **Evidence:** No `HEALTHCHECK` in `Dockerfile`. No `healthcheck:` block on the `app` service in `docker-compose.prod.yml`. The app already has `GET /api/health`, so the endpoint exists.
- **Fix:** Add to `docker-compose.prod.yml` under the `app` service:
  ```yaml
  healthcheck:
    test: ["CMD", "curl", "-f", "http://localhost:8080/api/health"]
    interval: 15s
    timeout: 5s
    retries: 3
    start_period: 10s
  ```
  Or add a `HEALTHCHECK` in the `Dockerfile`:
  ```dockerfile
  HEALTHCHECK --interval=15s --timeout=5s --retries=3 --start-period=10s \
    CMD curl -f http://localhost:8080/api/health || exit 1
  ```
  Note: the runtime image may not have `curl`; you could use `dotnet`-based health check or install `wget`.

### `INFRA-H003` â€” CI Workflow Has No Explicit Permissions
- **ID:** `INFRA-H003`
- **File:** `.github/workflows/test.yml`
- **Risk:** The workflow does not declare top-level `permissions:`. GitHub Actions defaults to `write-all` (unless the org has restricted this). This means a compromised dependency or malicious PR could use the token to push code, create releases, or modify issues.
- **Evidence:**
  ```yaml
  name: Test
  on:
    push:
      branches: [main]
    pull_request:
      branches: [main]
  jobs:
    test:
      runs-on: ubuntu-latest
      # No 'permissions' block
  ```
- **Fix:** Add least-privilege permissions at the top level:
  ```yaml
  permissions:
    contents: read
  ```

### `INFRA-M002` â€” CORS Allows Any Method and Any Header
- **ID:** `INFRA-M002`
- **File:** `fasolt.Server/Program.cs:L147-L159`
- **Risk:** Both development and production CORS policies use `.AllowAnyHeader().AllowAnyMethod()`. While origins are properly restricted, allowing all methods and headers in production is broader than necessary. A compromised allowed origin could send unexpected methods (e.g., `TRACE`) or headers.
- **Evidence:**
  ```csharp
  policy.WithOrigins(origins)
        .AllowCredentials()
        .AllowAnyHeader()
        .AllowAnyMethod();
  ```
- **Fix:** Restrict to the methods and headers actually used:
  ```csharp
  policy.WithOrigins(origins)
        .AllowCredentials()
        .WithHeaders("Content-Type", "Authorization", "X-Requested-With")
        .WithMethods("GET", "POST", "PUT", "DELETE");
  ```

### `INFRA-M003` â€” Missing Permissions-Policy and Cross-Origin Headers
- **ID:** `INFRA-M003`
- **File:** `fasolt.Server/Program.cs:L341-L348`
- **Risk:** The app sets `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, and `Content-Security-Policy` (good), but is missing `Permissions-Policy` and `Cross-Origin-Opener-Policy` headers. These provide additional defense-in-depth against speculative side-channel attacks (Spectre) and feature abuse.
- **Evidence:** The security headers middleware only sets four headers. No `Permissions-Policy` or `Cross-Origin-Opener-Policy`.
- **Fix:** Add these headers in the same middleware:
  ```csharp
  context.Response.Headers["Permissions-Policy"] = "camera=(), microphone=(), geolocation=()";
  context.Response.Headers["Cross-Origin-Opener-Policy"] = "same-origin";
  ```

### `INFRA-M004` â€” No Request Body Size Limit Configured
- **ID:** `INFRA-M004`
- **File:** `fasolt.Server/Program.cs`
- **Risk:** No explicit `RequestSizeLimit` or Kestrel `MaxRequestBodySize` is configured. The Kestrel default is ~28.6 MB. For an API that primarily ingests flashcard text, this is generous and could allow denial-of-service via large request bodies.
- **Evidence:** No `RequestSizeLimit`, `DisableRequestSizeLimit`, or `KestrelServerOptions.Limits.MaxRequestBodySize` found in the codebase.
- **Fix:** Configure a reasonable limit in `Program.cs`:
  ```csharp
  builder.WebHost.ConfigureKestrel(options =>
  {
      options.Limits.MaxRequestBodySize = 1_048_576; // 1 MB
  });
  ```

### `INFRA-M005` â€” Production Database Volume Uses Bind Mount
- **ID:** `INFRA-M005`
- **File:** `docker-compose.prod.yml:L34-L35`
- **Risk:** The production database stores data in a bind mount (`./db`). Unlike named volumes, bind mounts have the permissions of the host directory, which may be world-readable. A `docker compose down -v` does not affect bind mounts (good for safety, but the data is accessible to anyone with host file access). The dev compose correctly uses a named volume.
- **Evidence:**
  ```yaml
  volumes:
    - ./db:/var/lib/postgresql/data
  ```
- **Fix:** Use a named volume in production as well, or ensure the `./db` directory has restrictive permissions (`chmod 700`) and is excluded from backups that aren't encrypted. Named volume is preferred:
  ```yaml
  volumes:
    - pgdata:/var/lib/postgresql/data
  # ...
  volumes:
    pgdata:
  ```

### `INFRA-L001` â€” CI Uses Hardcoded Dev Credentials for Test Database
- **ID:** `INFRA-L001`
- **File:** `.github/workflows/test.yml:L17-L19`
- **Risk:** The CI postgres service uses `POSTGRES_PASSWORD: fasolt_dev`. While this is an ephemeral CI container (no real data), it establishes a pattern of embedding credentials directly in workflow files. If someone copies this pattern into a production pipeline, it becomes a real issue.
- **Evidence:**
  ```yaml
  env:
    POSTGRES_USER: fasolt
    POSTGRES_PASSWORD: fasolt_dev
  ```
- **Fix:** Acceptable for CI test databases. No change needed, but consider using `POSTGRES_HOST_AUTH_METHOD: trust` for CI (eliminates the password entirely for ephemeral containers) to avoid muscle-memory of embedding passwords.

### `INFRA-L003` â€” TrustAllProxies in Development appsettings
- **ID:** `INFRA-L003`
- **File:** `fasolt.Server/appsettings.Development.json:L12-L13`
- **Risk:** `TrustAllProxies: true` in development settings is fine. In production (`appsettings.Production.json:L15-L16`) it is also set to `true`, which the code already warns about at startup (Program.cs:L311-L315). The production setting is intentional per CLAUDE.md (app behind Cloudflare/Traefik, not directly reachable). However, if the deployment architecture changes to expose the app directly, IP spoofing via `X-Forwarded-For` becomes possible.
- **Evidence:** Both `appsettings.Development.json` and `appsettings.Production.json` have `"TrustAllProxies": true`.
- **Fix:** No immediate change needed given the documented architecture. The existing startup warning is good. Consider adding `KnownNetworks` configuration for production when the Docker network CIDR is known, to narrow trust.
