# Authentication & Authorization Review -- Findings

## Critical 🔴

### `AUTH-C001` -- Missing CSRF Protection on OAuth Login Form (POST)

- **ID:** `AUTH-C001`
- **File:** `fasolt.Server/Api/Endpoints/OAuthEndpoints.cs:L298-L313`
- **Risk:** The `/oauth/login` POST endpoint accepts a form submission with email, password, and returnUrl but has no CSRF token validation. An attacker can craft a malicious page that auto-submits a login form to this endpoint, performing a login CSRF attack. The attacker logs the victim into the *attacker's* account, then the victim unknowingly creates flashcards under the attacker's account via the MCP flow that follows.
- **Evidence:**
  ```csharp
  app.MapPost("/oauth/login", [AllowAnonymous] async (
      HttpContext context,
      SignInManager<AppUser> signInManager) =>
  {
      var form = await context.Request.ReadFormAsync();
      var email = form["email"].FirstOrDefault() ?? "";
      var password = form["password"].FirstOrDefault() ?? "";
      var returnUrl = form["returnUrl"].FirstOrDefault() ?? "/";
      // No CSRF token check
      var result = await signInManager.PasswordSignInAsync(email, password, ...);
  }).RequireRateLimiting("auth");
  ```
- **Fix:** Add an anti-forgery token to the login form. Generate a token in the GET handler and validate it in the POST handler. Alternatively, since this is a server-rendered form, use ASP.NET Core's `IAntiforgery` service:
  ```csharp
  // In GET handler: generate and embed token in the form
  var antiforgery = context.RequestServices.GetRequiredService<IAntiforgery>();
  var tokens = antiforgery.GetAndStoreTokens(context);
  // Add to HTML: <input type="hidden" name="__RequestVerificationToken" value="{{token}}" />

  // In POST handler: validate before processing
  var antiforgery = context.RequestServices.GetRequiredService<IAntiforgery>();
  if (!await antiforgery.IsRequestValidAsync(context))
      return Results.BadRequest("Invalid request");
  ```

### `AUTH-C002` -- Missing CSRF Protection on OAuth Consent Form (POST)

- **ID:** `AUTH-C002`
- **File:** `fasolt.Server/Api/Endpoints/OAuthEndpoints.cs:L393-L465`
- **Risk:** The `/oauth/consent` POST endpoint accepts a form submission with `client_id` and `decision` but has no CSRF token. An attacker can auto-submit a consent approval form while the victim has an active OAuth authorization flow, granting the attacker's OAuth client access to the victim's account. The encrypted cookie (`oauth_authorize_query`) provides some protection since it must be present, but a CSRF attack would send the victim's cookies automatically.
- **Evidence:**
  ```csharp
  app.MapPost("/oauth/consent", async (...) =>
  {
      var form = await context.Request.ReadFormAsync();
      var clientId = form["client_id"].FirstOrDefault() ?? "";
      var decision = form["decision"].FirstOrDefault() ?? "";
      // No CSRF token validation -- relies only on the encrypted cookie
  });
  ```
  The encrypted `oauth_authorize_query` cookie is `SameSite=Strict`, which does mitigate cross-site attacks in modern browsers. However, defense-in-depth requires explicit CSRF tokens, especially since SameSite support varies in older browsers and embedded webviews.
- **Fix:** Same approach as AUTH-C001 -- add anti-forgery tokens to the consent form. Additionally, verify the cookie's `SameSite=Strict` setting remains in place as a secondary defense.

## High 🟠

### `AUTH-H001` -- No Production Email Sender -- Password Reset/Email Change Non-functional

- **ID:** `AUTH-H001`
- **File:** `fasolt.Server/Program.cs:L125-L126`
- **Risk:** Only `DevEmailSender` exists, and it is only registered in development. In production, no `IEmailSender<AppUser>` is registered. This means the forgot-password and change-email endpoints will throw a runtime exception (`InvalidOperationException` for unresolved service) in production, or if a default no-op sender is injected by the framework, security tokens will be generated but never delivered -- users cannot reset passwords, and the generated tokens may leak in logs.
- **Evidence:**
  ```csharp
  if (builder.Environment.IsDevelopment())
      builder.Services.AddTransient<IEmailSender<AppUser>, DevEmailSender>();
  // No else branch -- production has no email sender registered
  ```
- **Fix:** Register a production email sender (e.g., via Resend, SendGrid, or SMTP). Alternatively, if email features are not yet ready for production, disable the forgot-password and change-email endpoints in non-development environments to avoid runtime errors.

### `AUTH-H002` -- OAuth Deny Redirects to Client-Controlled URI Without Validation

- **ID:** `AUTH-H002`
- **File:** `fasolt.Server/Api/Endpoints/OAuthEndpoints.cs:L460-L463`
- **Risk:** When a user denies consent, the server redirects to the first registered `redirect_uri` of the OAuth client. Since dynamic client registration allows any HTTPS URL as a redirect URI (line 31), an attacker who registers a malicious client can receive the user's browser redirect. Although the redirect only carries `error=access_denied` (no tokens leak), this is an open redirect -- the user's browser is sent to an attacker-controlled URL after interacting with a trusted fasolt page, enabling phishing.
- **Evidence:**
  ```csharp
  // Deny -- redirect to client with error
  var redirectUris = await applicationManager.GetRedirectUrisAsync(application);
  var clientRedirectUri = redirectUris.FirstOrDefault() ?? "/";
  var separator = clientRedirectUri.Contains('?') ? '&' : '?';
  return Results.Redirect($"{clientRedirectUri}{separator}error=access_denied");
  ```
- **Fix:** When denying consent, redirect back to a local error page (e.g., `/oauth/denied`) instead of the client's redirect URI. The client doesn't need to receive the deny callback since PKCE will time out. Alternatively, at minimum validate that the redirect URI was the one from the original authorize request, not just the first one from the database.

### `AUTH-H003` -- Dynamic Client Registration Has No Rate Limit Abuse Protection

- **ID:** `AUTH-H003`
- **File:** `fasolt.Server/Api/Endpoints/OAuthEndpoints.cs:L57-L115`
- **Risk:** The `/oauth/register` endpoint has rate limiting (`auth-strict`: 5/hour per IP), but an attacker with multiple IPs or behind a proxy can create unlimited OAuth client registrations. Each registration persists to the database. Over time, this fills the OpenIddict applications table, degrading performance and consuming storage. There is no cleanup mechanism for unused clients.
- **Evidence:**
  ```csharp
  app.MapPost("/oauth/register", [AllowAnonymous] async (...) =>
  {
      // ...
      await applicationManager.CreateAsync(descriptor);
      // No limit on total registrations, no expiry/cleanup
  }).RequireRateLimiting("auth-strict");
  ```
- **Fix:** Consider: (1) adding a global daily registration cap, (2) adding a TTL/expiry for unused client registrations, (3) requiring some form of proof-of-humanity for registration (e.g., a valid account). Alternatively, associate dynamic registrations with the requesting IP and cap registrations per IP in the database, not just per time window.

## Medium 🟡

### `AUTH-M001` -- OAuth Authorize Endpoint Missing Explicit Auth Requirement

- **ID:** `AUTH-M001`
- **File:** `fasolt.Server/Api/Endpoints/OAuthEndpoints.cs:L118`
- **Risk:** The `/oauth/authorize` GET endpoint does not have `[AllowAnonymous]` or `.RequireAuthorization()` explicitly. It manually calls `context.AuthenticateAsync()` and redirects to login if unauthenticated. While functionally correct, the lack of an explicit policy means the default authorization policy applies. Since the default policy accepts both cookie and OpenIddict bearer tokens, an OAuth bearer token could theoretically be used to approve authorization for another OAuth client -- a confused deputy issue.
- **Evidence:**
  ```csharp
  app.MapGet("/oauth/authorize", async (HttpContext context, ...) =>
  {
      var result = await context.AuthenticateAsync(IdentityConstants.ApplicationScheme);
      // Explicitly checks ApplicationScheme (cookie), so bearer tokens won't work here
  });
  ```
  **Mitigated** because the code explicitly authenticates against `IdentityConstants.ApplicationScheme` rather than using the request's ambient principal. However, the default authorization policy still runs and could short-circuit. This should be made explicit.
- **Fix:** Either mark the endpoint `[AllowAnonymous]` (since it handles its own auth check) or add a policy that requires cookie auth only, similar to `AdminCookieOnly`.

### `AUTH-M002` -- Lockout Duration Is Short (5 Minutes)

- **ID:** `AUTH-M002`
- **File:** `fasolt.Server/Program.cs:L39-L40`
- **Risk:** Account lockout is 5 minutes after 5 failed attempts. This allows sustained brute-force attacks with periodic 5-minute pauses. Combined with rate limiting (10/min per IP), an attacker can try ~50 passwords per lockout cycle.
- **Evidence:**
  ```csharp
  options.Lockout.MaxFailedAccessAttempts = 5;
  options.Lockout.DefaultLockoutTimeSpan = TimeSpan.FromMinutes(5);
  ```
- **Fix:** Increase lockout duration to 15-30 minutes. Consider progressive lockout (double the duration on each successive lockout). The rate limiter provides additional protection but operates per-IP and can be bypassed with distributed attacks.

### `AUTH-M003` -- DevEmailSender Logs Security Tokens

- **ID:** `AUTH-M003`
- **File:** `fasolt.Server/Infrastructure/Services/DevEmailSender.cs:L15-L29`
- **Risk:** The development email sender logs password reset links and email confirmation links at `Information` level. If this sender were accidentally used in production (see AUTH-H001), or if development logs are accessible to unauthorized parties, password reset tokens would be exposed in plaintext.
- **Evidence:**
  ```csharp
  public Task SendPasswordResetLinkAsync(AppUser user, string email, string resetLink)
  {
      _logger.LogInformation("Password reset link for {Email}: {Link}", email, resetLink);
      return Task.CompletedTask;
  }
  ```
- **Fix:** This is acceptable for development. Ensure the dev sender is never registered in production (see AUTH-H001 fix). Consider using `LogDebug` instead of `LogInformation` to reduce accidental exposure.

### `AUTH-M004` -- Frontend Admin Route Guard Is Client-Side Only

- **ID:** `AUTH-M004`
- **File:** `fasolt.client/src/router/index.ts:L68, L93-L97`
- **Risk:** The `/admin` route is guarded only by the frontend `isAdmin` check derived from `/api/account/me`. A user who modifies the frontend state or calls admin API endpoints directly could attempt privilege escalation. **Mitigated** because the backend admin endpoints use `RequireAuthorization("AdminCookieOnly")` which enforces the `Admin` role server-side.
- **Evidence:**
  ```typescript
  { path: '/admin', name: 'admin', component: () => import('@/views/AdminView.vue'), meta: { requiresAdmin: true } },
  // ...
  if (to.meta.requiresAdmin) {
      if (!auth.isAdmin) { return { name: 'study' } }
  }
  ```
- **Fix:** No backend change needed -- the server-side policy is correctly enforced. This is noted as a defense-in-depth observation. The frontend guard prevents UI confusion but is not a security boundary.

### `AUTH-M005` -- AllowAnyMethod in CORS Policy

- **ID:** `AUTH-M005`
- **File:** `fasolt.Server/Program.cs:L149, L159`
- **Risk:** Both development and production CORS policies use `.AllowAnyMethod()`. This allows `DELETE`, `PUT`, `PATCH`, and other methods from cross-origin requests. While CORS with credentials requires an explicit origin (which is set), allowing all methods is more permissive than necessary.
- **Evidence:**
  ```csharp
  policy.WithOrigins(origins)
        .AllowCredentials()
        .AllowAnyHeader()
        .AllowAnyMethod();  // Allows DELETE, PUT, PATCH, OPTIONS, etc.
  ```
- **Fix:** Restrict to methods actually used by the frontend: `.WithMethods("GET", "POST", "PUT", "DELETE")`.

## Low 🔵

### `AUTH-L001` -- Refresh Token Lifetime Is 14 Days Without Rotation

- **ID:** `AUTH-L001`
- **File:** `fasolt.Server/Program.cs:L87-L88`
- **Risk:** Refresh tokens are valid for 14 days. If a refresh token is compromised, an attacker has up to 14 days of access. OpenIddict supports refresh token rotation (issuing a new refresh token on each use and revoking the old one), which limits the window of compromise.
- **Evidence:**
  ```csharp
  options.SetAccessTokenLifetime(TimeSpan.FromHours(1))
         .SetRefreshTokenLifetime(TimeSpan.FromDays(14));
  ```
- **Fix:** Enable refresh token rotation in OpenIddict configuration. This is already the default behavior in OpenIddict for authorization code flows with PKCE, but verify it is active. Consider reducing the refresh token lifetime to 7 days for higher-security environments.

### `AUTH-L002` -- Registration Endpoint May Leak Email Existence

- **ID:** `AUTH-L002`
- **File:** ASP.NET Core Identity's built-in `/api/identity/register` endpoint
- **Risk:** The built-in Identity `register` endpoint returns a `400 Bad Request` with a `DuplicateUserName` error when an email is already registered. This allows account enumeration. **Partially mitigated** by rate limiting (`auth` policy: 10/min per IP). Note: the custom `forgot-password` endpoint correctly returns 200 regardless (line 107 of AccountEndpoints.cs).
- **Evidence:** This is standard ASP.NET Core Identity behavior. The iOS app already parses duplicate-email errors to show "An account with this email already exists."
- **Fix:** This is a known trade-off in ASP.NET Core Identity. Mitigations include: (1) rate limiting (already present), (2) CAPTCHA on registration, (3) returning a generic success message and sending a "you already have an account" email instead. Option 3 would require overriding the default Identity registration endpoint.

### `AUTH-L003` -- Cookie ExpireTimeSpan of 1 Day with Sliding Expiration

- **ID:** `AUTH-L003`
- **File:** `fasolt.Server/Program.cs:L107-L108`
- **Risk:** Session cookies expire after 1 day but have sliding expiration, meaning active users remain logged in indefinitely. This is acceptable for a flashcard app but worth noting -- there is no absolute session timeout.
- **Evidence:**
  ```csharp
  options.ExpireTimeSpan = TimeSpan.FromDays(1);
  options.SlidingExpiration = true;
  ```
- **Fix:** Consider adding an absolute session lifetime (e.g., 30 days) to force re-authentication periodically. This can be done by tracking session creation time and checking it in the `OnValidatePrincipal` event.

## What's Done Well

- **Consistent tenant isolation**: All service methods filter by `userId` in every query. No endpoint allows accessing another user's data. The `CardService`, `DeckService`, `SearchService`, `SourceService`, and `OverviewService` all properly scope data to the authenticated user.
- **Admin endpoints use a dedicated cookie-only policy** (`AdminCookieOnly`): OAuth bearer tokens cannot access admin endpoints, preventing escalation through MCP/OAuth flows.
- **PKCE required for OAuth**: `options.RequireProofKeyForCodeExchange()` enforces PKCE, protecting the authorization code flow from interception.
- **Cookie security settings are solid**: `HttpOnly = true`, `SameSite = Strict`, `SecurePolicy = Always` in production. These settings effectively prevent cookie theft via XSS and mitigate most CSRF scenarios for the SPA API endpoints.
- **Rate limiting on auth endpoints**: Login, registration, and client registration all have rate limits.
- **ForgotPassword prevents enumeration**: Returns 200 regardless of whether the email exists.
- **MCP auth uses McpUserResolver consistently**: Every MCP tool resolves the user from the HTTP context, and the `/mcp` endpoint requires authorization.
- **Open redirect prevention on OAuth login**: `IsLocalUrl()` validates returnUrl, rejecting absolute URLs and protocol-relative URLs (`//`).
- **Data protection keys persisted to DB**: Prevents key loss across deploys.
- **Development certificates gated by environment**: Non-development environments must provide real certificates or the app throws.
- **iOS app uses Keychain with `kSecAttrAccessibleWhenUnlockedThisDeviceOnly`**: Tokens are hardware-bound and encrypted at rest.
- **Security headers present**: X-Content-Type-Options, X-Frame-Options (DENY), CSP, Referrer-Policy all set.
- **OAuth consent flow uses encrypted cookie for state**: The `oauth_authorize_query` cookie is encrypted with data protection, preventing tampering with the authorization parameters.
