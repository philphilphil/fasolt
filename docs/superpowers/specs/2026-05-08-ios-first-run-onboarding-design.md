# iOS first-run onboarding

**Status:** spec
**Date:** 2026-05-08
**Scope:** iOS app (`fasolt.ios/`)

## Problem

A new iOS user signs up and is dropped into an empty dashboard. The only visible cue is a "Create demo deck" button. Nothing tells them that Fasolt's value comes from connecting an AI client (Claude, ChatGPT, Copilot, etc.) via MCP, and the MCP setup itself is buried in Settings → MCP Setup. Users who don't know to look there see an empty app and assume there's nothing more to it.

## Goal

After first sign-in, a new user should understand:

1. What Fasolt does (spaced repetition, cards generated by their AI).
2. That an AI client is required for the app to be useful long-term.
3. How to connect that client.

A user who skips the explanation should still be funneled toward setup via a persistent empty-state cue on the Dashboard, and should be able to re-open the welcome flow from Settings.

## Non-goals

- No interactive product tour beyond the welcome flow.
- No server-side onboarding state — the welcome flow is per-device.
- No verification that MCP is actually connected. The empty-state cue keys off "do you have any cards yet," which is a sufficient proxy.
- No changes to the existing sign-up / sign-in screen (`OnboardingView.swift` — misleadingly named, but out of scope).
- No copy changes to `McpSetupSection.swift`. It is reused as-is.

## Design

### Trigger

A `@AppStorage("hasSeenWelcomeFlow")` Bool, default `false`, lives in `FasoltApp.swift`. After authentication, if the flag is `false`, `WelcomeFlowView` is presented as a `.fullScreenCover`. The flag flips to `true` on either "Done" (step 4) or "Skip for now" (any step). Per-device by design — seeing a 30-second intro again on a new device is acceptable, and avoids a server round-trip on every cold launch.

### Welcome flow

A new view `Views/Onboarding/WelcomeFlowView.swift` wraps a `TabView` with `.page` style for swipe-between-steps, plus a "Skip for now" button in the nav bar and a "Continue / Done" primary button at the bottom.

**Step 1 — How Fasolt works**

- Hero: Fasolt logo.
- Headline: *Spaced repetition, powered by your AI.*
- Bullets (icon + text):
  - 🤖 *Your AI creates flashcards — from your notes, or from scratch.*
  - 🔁 *Cards sync to Fasolt automatically.*
  - 🧠 *You review them here at the right time, the right amount.*

**Step 2 — What you need**

- Headline: *You'll need an AI client.*
- Sub: *Fasolt connects to Claude, ChatGPT, GitHub Copilot, or any MCP-compatible client. Your AI does the work of creating and updating cards.*
- Visual: row of client labels/logos (Claude, ChatGPT, Copilot).
- Footnote: *Don't have one yet? You can still try a demo deck after this.*

**Step 3 — Connect your AI**

- Headline: *Add Fasolt to your AI client.*
- Embeds the existing `McpSetupSection` inside a `List` — gives users the MCP URL with copy buttons and per-client instructions (Claude Code, Claude.ai web, ChatGPT, Copilot CLI).
- Bottom button: **Continue**.

**Step 4 — You're set**

- Big checkmark.
- Headline: *You're ready to go.*
- Single CTA: **Open Dashboard** — sets `hasSeenWelcomeFlow = true` and dismisses the cover.

### Empty-state restructure (DashboardView)

When `viewModel.totalCards == 0 && !viewModel.isLoading && viewModel.errorMessage == nil`, the current demo-deck block is replaced with:

- A card-styled prompt:
  - 🤖 icon
  - Title: *Connect your AI*
  - Body: *Add Fasolt to Claude, ChatGPT, or another MCP client to start creating cards.*
  - Primary button: **Set up now** — opens a sheet containing `List { McpSetupSection(serverURL:) }`.
- Below the card, a smaller text link: **Just try a demo deck** — invokes the existing `viewModel.createDemoDeck()`.

Once `totalCards > 0` the block disappears, exactly as today.

The `serverURL` for `McpSetupSection` comes from `authService.serverURL` (already used elsewhere; it's the same source the Settings screen uses).

### Settings entry point

In `SettingsView.swift`, add a row labelled **Show welcome again**. Tapping it sets `@AppStorage("hasSeenWelcomeFlow")` to `false`. The existing `.fullScreenCover` binding in `FasoltApp` re-fires and presents the welcome flow.

Place the row in the same area as the existing `McpSetupSection` (e.g. immediately above it).

## File changes

**New**
- `fasolt.ios/Fasolt/Views/Onboarding/WelcomeFlowView.swift` — the 4-step `TabView` and step subviews. Single file is fine; if step views grow, split into `WelcomeFlowStep.swift`.

**Modified**
- `fasolt.ios/Fasolt/FasoltApp.swift` — add `@AppStorage("hasSeenWelcomeFlow")`; add `.fullScreenCover` presenting `WelcomeFlowView` when authenticated and flag is `false`.
- `fasolt.ios/Fasolt/Views/Dashboard/DashboardView.swift` — replace the existing empty-state `VStack` (lines ~20–44) with the new "Connect your AI" prompt; add `@State` for the MCP sheet.
- `fasolt.ios/Fasolt/Views/Settings/SettingsView.swift` — add "Show welcome again" row.

**Reused unchanged**
- `fasolt.ios/Fasolt/Views/Settings/McpSetupSection.swift` — embedded inside a `List` in step 3 of the welcome flow and inside the Dashboard's "Set up now" sheet.

## Backend / API

None. No new endpoints, no schema changes.

## Testing

Manual, via iOS simulator with a fresh dev account:

1. **Happy path** — register a new user → welcome cover appears → swipe through 4 steps → tap **Open Dashboard** → empty-state shows "Connect your AI" prompt → tap "Just try a demo deck" → cards appear → prompt disappears.
2. **Skip path** — register → tap "Skip for now" on step 1 → cover dismisses → empty-state still shows the same prompt → tap "Set up now" → sheet appears with MCP URL and per-client instructions → dismiss.
3. **Re-launch** — Settings → "Show welcome again" → cover reappears → complete or skip → flag persists across cold launch.
4. **Existing user upgrading the app** — sign in as a user that already has cards. The `hasSeenWelcomeFlow` flag is `false` on the upgraded install, so the welcome flow does appear once. Verify it dismisses cleanly, the dashboard then shows the user's normal data (no empty-state prompt because `totalCards > 0`), and the flag persists.

No new unit tests — pure SwiftUI view glue, no business logic. The FSRS engine, API client, and view models are untouched.

## Risks

- **Existing users see the welcome flow once after upgrading.** Acceptable: it's short, dismissible, and the content is still relevant ("here's how to set up MCP if you haven't"). Could be mitigated by checking `totalCards > 0` once at first launch and pre-flipping the flag, but this adds complexity for a one-time event.
- **`McpSetupSection` is currently a `Section`-shaped view (must live inside a `List`).** We embed `List { McpSetupSection(...) }` in step 3 and in the dashboard sheet. Acceptable; visually consistent with Settings.
- **Server URL accuracy.** `authService.serverURL` is the URL the user authenticated against, which is the right MCP URL to display. No new state needed.
