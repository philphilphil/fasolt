# Remote MCP Server with OAuth

## Goal

Add a remote MCP endpoint to the existing `fasolt.Server` so users can connect their AI client with a single URL — no .NET SDK, no tool install, no token copying. Keep the local stdio MCP server for advanced use cases.

## Decisions

- Remote MCP endpoint at `/mcp` in `fasolt.Server` using Streamable HTTP transport
- OAuth 2.1 via OpenIddict (not hand-rolled) — backend acts as its own authorization server
- OAuth for remote MCP, API tokens for local MCP — each path has one auth mechanism
- Local stdio MCP server (`fasolt.Mcp/`) stays as-is for advanced users
- MCP setup page updates: remote-first, local setup in collapsible "Advanced" section

## Architecture

### Transport

The `ModelContextProtocol.AspNetCore` NuGet package provides Streamable HTTP server transport:

```csharp
builder.Services.AddMcpServer()
    .WithHttpTransport()
    .AddAuthorizationFilters()
    .WithTools<CardTools>()
    .WithTools<DeckTools>()
    .WithTools<SourceTools>();

app.MapMcp("/mcp");
```

`MapMcp()` handles both Streamable HTTP and legacy SSE automatically.

### Tool Implementations

The 6 MCP tools (CreateCards, SearchCards, ListCards, ListSources, ListDecks, CreateDeck) currently live in `fasolt.Mcp/Tools/` and call the REST API via `ApiClient`. For the remote server, these tools run inside `fasolt.Server` and call services directly — no HTTP round-trip.

Create new tool classes in `fasolt.Server/Api/McpTools/` that use the existing application services (CardService, DeckService, etc.) directly. These are thin wrappers with MCP attributes.

The tools resolve the current user from the HTTP context (OAuth token → user identity).

### OAuth 2.1 via OpenIddict

Use OpenIddict as the OAuth authorization server. It integrates with ASP.NET Core Identity and EF Core, handles JWT issuance, PKCE validation, token refresh, and metadata endpoints.

#### NuGet Packages

```xml
<PackageReference Include="OpenIddict.AspNetCore" Version="7.0.0" />
<PackageReference Include="OpenIddict.EntityFrameworkCore" Version="7.0.0" />
<PackageReference Include="ModelContextProtocol.AspNetCore" Version="1.0.0" />
```

#### OpenIddict Configuration (Program.cs)

```csharp
builder.Services.AddOpenIddict()
    .AddCore(options =>
    {
        options.UseEntityFrameworkCore()
               .UseDbContext<AppDbContext>();
    })
    .AddServer(options =>
    {
        options.SetAuthorizationEndpointUris("/oauth/authorize")
               .SetTokenEndpointUris("/oauth/token");

        options.AllowAuthorizationCodeFlow()
               .AllowRefreshTokenFlow();

        options.RequireProofKeyForCodeExchange(); // PKCE mandatory

        // Dev certs for development, real certs for production
        options.AddDevelopmentEncryptionCertificate()
               .AddDevelopmentSigningCertificate();

        options.UseAspNetCore()
               .EnableAuthorizationEndpointPassthrough()
               .EnableTokenEndpointPassthrough();
    })
    .AddValidation(options =>
    {
        options.UseLocalServer();
        options.UseAspNetCore();
    });
```

#### Required Endpoints

OpenIddict handles most of these automatically. We implement the passthrough handlers:

1. **Authorization Server Metadata** — `GET /.well-known/openid-configuration`
   - Auto-generated by OpenIddict (RFC 8414 compatible)

2. **Protected Resource Metadata** — `GET /.well-known/oauth-protected-resource`
   - Manual minimal API endpoint returning JSON with `resource` URL and `authorization_servers` array
   - Required by MCP spec (RFC 9728)

3. **Authorization Endpoint** — `GET /oauth/authorize` (passthrough)
   - If user not authenticated → redirect to login page
   - If authenticated → show consent screen (or auto-approve for first-party)
   - Issue authorization code and redirect to client callback
   - Minimal API handler that builds claims and calls `Results.SignIn()` with OpenIddict scheme

4. **Token Endpoint** — `POST /oauth/token` (passthrough)
   - OpenIddict handles authorization code exchange, PKCE validation, refresh token rotation
   - Minimal API handler resolves the user and builds the claims principal

5. **Dynamic Client Registration** — `POST /oauth/register`
   - Manual endpoint using OpenIddict's application manager to create clients
   - Accepts `client_name`, `redirect_uris`, returns `client_id`
   - MCP spec says SHOULD support (RFC 7591)

#### OAuth Flow

1. MCP client sends request to `/mcp` without token
2. Server returns `401` with `WWW-Authenticate` header pointing to resource metadata
3. Client fetches `/.well-known/oauth-protected-resource`
4. Client fetches `/.well-known/openid-configuration`
5. Client registers via `/oauth/register` (if not already registered)
6. Client opens browser → `/oauth/authorize?response_type=code&client_id=...&code_challenge=...&resource=...`
7. User logs in (existing ASP.NET Identity login) and approves
8. Server redirects to client callback with authorization code
9. Client exchanges code for access token at `/oauth/token`
10. Client uses access token as `Authorization: Bearer <token>` on `/mcp`

#### Token Format

- Access tokens: JWT signed by OpenIddict (signing certificates)
- Short-lived (1 hour, configurable via OpenIddict)
- Refresh tokens: opaque, managed by OpenIddict, stored in DB, rotated on use
- Claims: `sub` (user ID), `name`, standard OpenIddict claims

### Database Changes

OpenIddict manages its own tables via EF Core. Adding `options.UseOpenIddict()` to the `DbContext` configuration adds:

- **OpenIddictApplications** — registered OAuth clients (dynamic registration)
- **OpenIddictAuthorizations** — authorization grants
- **OpenIddictScopes** — OAuth scopes
- **OpenIddictTokens** — access tokens, refresh tokens, authorization codes

No custom OAuth tables needed — OpenIddict handles all of this.

A single EF Core migration adds all four tables.

### Auth Middleware Stack

The server needs to support three auth methods simultaneously:

1. **Cookie auth** — existing, for web UI
2. **API token auth** — existing, for REST API and local MCP (`Authorization: Bearer sm_...`)
3. **OpenIddict token validation** — new, for remote MCP (`Authorization: Bearer eyJ...`)

The existing `BearerTokenHandler` checks for `sm_` prefix. OpenIddict JWT tokens start with `eyJ` (base64 JWT header). The middleware can dispatch based on token format, or use ASP.NET Core's multi-scheme authentication with a policy selector.

### Security Requirements (per MCP spec)

- PKCE mandatory (S256) — `RequireProofKeyForCodeExchange()`
- Resource parameter (RFC 8707) — validate in authorize/token handlers
- Token audience validation — OpenIddict handles this
- HTTPS required for all OAuth endpoints in production
- Redirect URI exact match validation — OpenIddict default behavior
- Refresh token rotation — OpenIddict handles this
- Short-lived access tokens (1 hour)

### Login Page for OAuth Flow

When the OAuth authorize endpoint is hit and the user isn't logged in, they need to authenticate. Two options:

- **Redirect to existing SPA login** (`/login`) — but the SPA login does a cookie-based API call and redirects to dashboard, not back to the OAuth flow
- **Server-rendered login page** — a minimal Razor page or static HTML form at `/oauth/login` that posts credentials, sets the auth cookie, and redirects back to `/oauth/authorize`

The server-rendered approach is simpler for OAuth flows since it avoids SPA routing complexity. It's a single form (email + password) that posts to the existing Identity login endpoint, then redirects.
